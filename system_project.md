# SystemIO

## 1. Цель
SystemIO — персональный мессенджер / система обмена сообщениями с веб-интерфейсом, серверным API и возможностью дальнейшего расширения.

## 2. MVP
Минимальная рабочая версия должна уметь:
- регистрировать и авторизовать пользователя
- хранить пользователей в БД
- отправлять и получать текстовые сообщения
- показывать список диалогов
- отображать историю сообщений
- обновлять чат без перезагрузки страницы

## 3. Архитектура

### Клиент
- HTML/CSS/JavaScript
- страница входа
- список диалогов
- окно чата
- форма отправки сообщения

### Сервер
- Python Flask
- REST API для авторизации, пользователей и истории сообщений
- WebSocket для новых сообщений в реальном времени

### База данных
- MySQL
- таблицы пользователей, диалогов, сообщений, сессий

## 4. Поток работы
1. Пользователь входит в систему.
2. Клиент получает список диалогов.
3. При открытии диалога загружается история сообщений.
4. Новое сообщение отправляется на сервер.
5. Сервер сохраняет сообщение в БД.
6. Сервер отправляет событие получателю через WebSocket.
7. Клиент обновляет чат без перезагрузки.

## 5. Структура БД

### users
- id BIGINT PK
- login VARCHAR(64) UNIQUE
- password_hash VARCHAR(255)
- display_name VARCHAR(128)
- created_at DATETIME
- last_seen DATETIME
- is_active TINYINT(1)

### dialogs
- id BIGINT PK
- type ENUM('private','group')
- title VARCHAR(255) NULL
- created_at DATETIME
- created_by BIGINT

### dialog_members
- id BIGINT PK
- dialog_id BIGINT
- user_id BIGINT
- joined_at DATETIME
- role ENUM('member','admin')

### messages
- id BIGINT PK
- dialog_id BIGINT
- sender_id BIGINT
- body TEXT
- created_at DATETIME
- edited_at DATETIME NULL
- status ENUM('sent','delivered','read')

### user_sessions
- id BIGINT PK
- user_id BIGINT
- session_token VARCHAR(255) UNIQUE
- ip_address VARCHAR(64)
- user_agent TEXT
- created_at DATETIME
- expires_at DATETIME

## 6. API MVP

### POST /api/auth/register
Создание пользователя.

### POST /api/auth/login
Вход в систему.

### POST /api/auth/logout
Выход из системы.

### GET /api/me
Текущий пользователь.

### GET /api/dialogs
Список диалогов.

### GET /api/dialogs/{id}/messages
История сообщений.

### POST /api/dialogs/{id}/messages
Отправка сообщения.

### GET /api/users/search?q=
Поиск пользователя.

## 7. WebSocket события
- message:new
- message:delivered
- message:read
- user:online
- user:offline

## 8. Безопасность
- пароли хранить только в виде хеша
- использовать bcrypt или argon2
- проверять права доступа к диалогу
- ограничить частоту запросов login / send message
- валидировать входные данные
- экранировать вывод сообщений в интерфейсе

## 11. Заметка по архитектуре (отложено)
В ходе обсуждения была предложена более модульная архитектура backend из ~8 модулей (app, api, services, models, database, realtime, utils, tests). Решено **пока не усложнять MVP** и оставить текущую упрощённую структуру проекта.

К этой архитектуре планируется вернуться позже, после стабилизации базового функционала:
- регистрация и авторизация
- список диалогов
- отправка и получение сообщений
- история сообщений

Когда MVP станет стабильным, можно будет создать отдельную ветку разработки (например `architecture_v2`) и постепенно перенести код на модульную структуру.

## 15. Architecture Diagram

Простая схема взаимодействия компонентов SystemIO.

```text
        Browser
   (HTML / CSS / JS)
            │
            │ HTTP / JSON API
            ▼
        Flask Server
        (backend/app.py)
            │
            │ SQL queries
            ▼
        SQLite Database
        (systemio.db)
```

### Поток сообщения

```text
User A (browser)
      │
      │ POST /api/dialogs/{id}/messages
      ▼
Flask API
      │
      │ сохранить сообщение
      ▼
SQLite
      │
      │ GET /messages?after_id=
      ▼
User B (browser polling)
```

### Текущая модель работы

- клиент отправляет HTTP запросы к API
- сервер обрабатывает логику и сохраняет данные
- база SQLite хранит пользователей, диалоги и сообщения
- клиент периодически запрашивает новые сообщения (polling)

В будущих версиях polling может быть заменён на WebSocket realtime‑канал.
